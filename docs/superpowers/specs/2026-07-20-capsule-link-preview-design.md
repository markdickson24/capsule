# Capsule Invite Link Preview — Design

## Problem

Every capsule invite — the "Share link" button, the QR code, Onboarding's
"Invite people" — hands out `capsule://join/<id>`, a custom URL scheme.
Messaging apps (iMessage, WhatsApp, Slack, Discord, ...) only unfurl
`http(s)://` URLs into rich preview cards; a custom scheme never gets a
preview at all. Today, sharing a capsule produces a bare line of text with
no image, no title, nothing.

## Goal

Sharing a capsule produces a rich, personalized preview card (title, short
description, branded image with the capsule's name on it) in every major
messaging app, and tapping the link either opens the app directly (if
installed) or lands on the landing page with context, instead of doing
nothing.

## Scope

**In scope:**
- One new URL format, `https://getcapsuleapp.com/join/<id>`, replacing
  `capsule://join/<id>` everywhere it's generated by the client.
- A Netlify Edge Function serving that URL: real Open Graph tags in the
  initial HTML response (crawlers don't execute JS), a personalized PNG
  image, and a JS-based redirect into the installed app with a graceful
  fallback to the landing page.
- Updating all three generation sites (share-sheet message ×2, QR code) and
  the in-app QR *scanner* to recognize the new format.
- A small "you were invited" banner on the landing page for the fallback
  case.

**Explicitly out of scope (deferred):**
- **Real Apple Universal Links / Android App Links** (`apple-app-site-association`,
  digital asset links) — these give a true one-tap open with no visible
  redirect page, but need a stable App Store presence and more Apple/Google
  console setup to be worth doing correctly. The JS-redirect pattern used
  here is standard practice (Slack, Notion, etc. all do this) and works
  today without any of that. Worth revisiting once the app is actually on
  the App Store.
- **Custom font embedding in the generated image.** The renderer's built-in
  fonts are used for v1 (see Image Generation below) instead of the site's
  actual brand fonts (Fraunces/Hanken Grotesk) — a cosmetic gap, not a
  functional one, and cheap to upgrade later without touching the rest of
  the pipeline.
- **Search-engine indexing of `/join/<id>` pages.** These are private invite
  links, not public content — the generated page ships `noindex` and is
  deliberately never added to `sitemap.xml`.

## Architecture

```
Share/QR                     https://getcapsuleapp.com/join/<id>
   │                                        │
   │                          ┌─────────────┴─────────────┐
   │                          │   Netlify Edge Function     │
   │                          │   netlify/edge-functions/   │
   │                          │   join.ts                   │
   │                          └─────────────┬─────────────┘
   │                                        │
   │                    calls capsule_join_preview RPC
   │                    (service-role key, server-side only)
   │                                        │
   │                          ┌─────────────┴─────────────┐
   │                          │ path ends in /image?        │
   │                          ├─────────────┬─────────────┤
   │                          │ yes          │ no            │
   │                          ▼              ▼               │
   │                   build SVG card   render HTML page
   │                   → resvg render   with OG tags pointing
   │                   → PNG response   at the /image URL,
   │                                    + JS redirect logic
   │                                        │
   │                          ┌─────────────┴─────────────┐
   │                          │  window.location =           │
   │                          │  capsule://join/<id>          │
   │                          │  (app installed → opens)      │
   │                          │  timeout → redirect to        │
   │                          │  getcapsuleapp.com?invited...  │
   └──────────────────────────┴────────────────────────────┘
```

### One function, not two

`/join/<id>` (the page) and `/join/<id>/image` (the PNG) are both handled
by a **single** edge function registered at `path = "/join/*"`, branching
internally on whether the path ends in `/image`. Netlify's edge function
path matching uses `URLPattern`, where a wildcard segment can overlap
across multiple registered functions in ways that are easy to get wrong
(both could fire on the same request unless one explicitly halts the
chain). One function with an internal branch sidesteps that class of bug
entirely — simpler and more predictable than two functions with adjacent
wildcard paths.

### Directory placement (a real footgun, resolved)

The current `netlify.toml` sets `base = "landing"`. Netlify resolves the
default edge-functions directory *relative to `base`* — meaning if the
function lived at `landing/netlify/edge-functions/`, its source file would
sit **inside the publish directory** and get served as a static file at a
public URL (Netlify's own docs flag this exact mistake). `base` was
protecting against Netlify auto-running the RN app's build (`expo export -p
web`) during framework auto-detection — a problem we already fixed by
clearing the dashboard's build command, so `base` is no longer load-bearing
for that. This design drops it: `publish = "landing"` (root-relative, still
unambiguous) and the edge function lives at repo-root
`netlify/edge-functions/join.ts` — a sibling to `landing/` and
`supabase/functions/`, structurally outside the publish path.

### Data source: service-role key, not the public anon path

`capsule_join_preview(p_capsule_id)` already returns exactly what's needed
— `title`, `owner_name`, `owner_avatar`, `member_count` — and is already
the mechanism `QRScannerScreen` uses for the in-app pre-join preview. That
in-app call goes through the authenticated Supabase client and is
unaffected by this change. The **edge function** is a different caller: an
unauthenticated web request from an arbitrary crawler or browser. Rather
than depending on `anon`-level PostgREST access to this RPC (which was
deliberately revoked in the July 18 launch-hardening pass — see
`20260718120000_revoke_anon_rpc_execute.sql`), the edge function calls it
server-side using the Supabase **service-role key**, held only as a
Netlify environment variable (Functions scope — never shipped to any
client, never in `netlify.toml`). This is a *more* controlled posture than
making the RPC anon-callable again: "owner name + avatar, given just a
UUID" stays gated behind our own function instead of being permanently
public over PostgREST.

### Image generation

A hand-built SVG string (1200×630 — the universal OG-image size), rendered
to PNG via [`resvg_wasm`](https://deno.land/x/resvg_wasm) — a pure-WASM
Deno port of the Rust `resvg` renderer, no native bindings (required: edge
runtimes like Netlify's don't support native addons), no filesystem
access, three fonts built in. Usage is a single call:

```ts
import { render } from "https://deno.land/x/resvg_wasm@0.2.0/mod.ts";
const png: Uint8Array = await render(svgString);
```

Card content: dark background (`#0A0A0A`) with a soft accent-color radial
glow, a hand-drawn lock icon (SVG paths — no icon font needed), the
capsule title (large, bold, wrapped to at most 2 lines with ellipsis
truncation — SVG `<text>` doesn't auto-wrap, so line-breaking is done in
JS before building the markup), "`{owner_name}` invited you" in the accent
color, and "`{member_count}` people already in" in muted gray. The owner's
avatar is embedded as a `<image>` with a base64 data URI on a best-effort
basis (fetched with a short timeout; on any failure — missing avatar, slow
fetch — it's silently skipped in favor of a plain circular initial-letter
badge, never blocking the response).

### Redirect logic

The HTML page's `<body>` has no visible content beyond a loading state —
it's meant to be invisible in the common case. Inline script:

```js
window.location.href = "capsule://join/<id>";
setTimeout(() => {
  window.location.href = "https://getcapsuleapp.com/?invited_by=<owner_name>&capsule=<title>";
}, 1200);
```

If the app is installed, the OS intercepts the custom-scheme navigation
immediately and the page is backgrounded before the timeout fires. If not,
the timeout redirects to the landing page with query params a small
`site.js` snippet reads to show a contextual banner ("👋 `{owner}` invited
you to a Capsule — join the waitlist below for early access") instead of
the generic hero.

## App-side changes

Four call sites, all mechanical — same URL, different string template:

- `CapsuleDetailScreen.shareLink()` — share-sheet message
- `CapsuleDetailScreen`'s QR code `value` prop
- `OnboardingScreen`'s step-5 "Invite people" share message
- `QRScannerScreen.handleScan()` — the regex gains a second alternative so
  it accepts **both** `capsule://join/<id>` and
  `https://getcapsuleapp.com/join/<id>`. Not a real backward-compat need
  (QR codes are generated fresh each time, never pre-printed), but free
  insurance against any already-open share sheet or copied link from
  before this ships.

`useDeepLinks.ts` (the app's own `capsule://` handler) needs **no
changes** — it only ever receives `capsule://` URLs from the OS; the
`https://getcapsuleapp.com/join/<id>` page never reaches app code, only
the browser.

## Testing approach

No automated test can verify how iMessage, WhatsApp, Slack, and Discord
actually render an unfurl — their crawlers differ in caching, image
format tolerance, and whether they even attempt a fetch over
non-production domains. Verification is manual and explicit:
1. `curl` the page and image URLs directly, confirm the OG tags and a
   valid PNG come back.
2. A generic OG-tag debugger (e.g. a Twitter/X card validator or
   metatags.io) to sanity-check the tags parse as intended.
3. Actually send the link to yourself in iMessage, WhatsApp, and Slack (or
   whichever you use) and look at the resulting card.
4. Confirm the redirect: tap the link on a phone with the app installed
   (opens app), and on one without (or with the app force-quit and the
   scheme handler unregistered, if that's feasible to simulate — otherwise
   note as a known gap) to confirm the landing-page fallback and banner.

If step 3 reveals a crawler that mishandles the image (some are stricter
about PNG color profiles or dimensions than others), the fallback is
narrowing the SVG/render pipeline, not the overall architecture — the
edge-function/redirect design doesn't change.

## Self-review

- **Placeholders:** none — every code snippet above is the real API,
  verified against Netlify's and the resvg-wasm-for-Deno docs directly
  rather than assumed.
- **Consistency:** the edge function's Supabase-calling style
  (`supabase-js` via `esm.sh`, not raw `fetch`) matches every other Deno
  function in this repo, even though it lives in Netlify rather than
  Supabase — deliberate, for codebase consistency.
- **Scope:** tightly bounded — no DB migration needed (the RPC already
  returns everything required), no changes to `useDeepLinks.ts` or the
  app's native deep-link handling at all.
- **Ambiguity resolved during design, not left for the plan to trip on:**
  the two-edge-function wildcard-overlap risk (→ single function with
  internal branch) and the `base`/publish-directory footgun (→ dropped
  `base`, function moved to repo root) were both caught and fixed here
  rather than discovered mid-implementation.
