# Capsule — Performance Audit

Bottlenecks found in a full read-through of the client, edge functions, and cron jobs,
cross-checked against the **live** database (indexes, `pg_proc` volatility, `cron.job`
definitions, and Supabase's performance advisors). Ordered by ROI: measurable user-facing
or cost impact per unit of effort. **No code has been changed** — this is a recommendation
list.

A note on scale: the DB tables are small today, so pure-SQL wins are mostly cheap
insurance. The big, *measurable-right-now* wins are all on the media pipeline — bytes
over the radio dominate everything else in this app by orders of magnitude.

Legend: 🔴 High ROI · 🟠 Medium · 🟡 Cheap insurance / at-scale

---

## 🔴 1. Library/camera-picker photos upload at full resolution — ~10× more bytes than needed

**File:** `src/screens/app/CapsuleDetailScreen.tsx` — `pickFromLibrary` (~line 1404),
`pickFromCamera` (~line 1418), both feeding `uploadPhotos`

```ts
const result = await ImagePicker.launchImageLibraryAsync({
  mediaTypes: ImagePicker.MediaTypeOptions.Images,
  allowsMultipleSelection: true,
  quality: 0.8,          // ← compression only, NO resize
});
```

`quality: 0.8` re-encodes but does **not** downscale. A modern iPhone photo is
4000×3000+; at q0.8 that's typically **2–5 MB per photo**. Compare `CameraScreen`'s
`processPhoto`, which resizes to 1920px wide first (~300–600 KB). So the most common
add-photos path (library multi-select) uploads roughly **5–10× more bytes per photo**
than the in-app camera path, and those full-res originals are then what every member
downloads in the full-screen viewer (the grid is saved by `transformMediaUrl`, but the
viewer uses the raw `signedUrl`).

Same gap in **`PreviewScreen.uploadFile`** (~line 30): media arriving via the share
intent (`useShareIntent` → `Preview`) is uploaded as-is with no resize — sharing straight
from Photos bypasses the 1920px pipeline entirely.

**Recommendation:** run picked/shared photos through the exact `ImageManipulator`
resize-to-1920 step `CameraScreen.processPhoto` already uses, before upload. One shared
helper, applied at three call sites.

**Measure:** `media.size_bytes` is already recorded — compare the average for
camera-captured vs. library-picked rows before/after. Expect upload wall-time and
storage growth to drop roughly in proportion (5–10×), and the biggest single
improvement to "add 10 photos" UX on cellular.

## 🔴 2. Video grid thumbnails are generated by *downloading the video* on every member's device

**File:** `src/screens/app/CapsuleDetailScreen.tsx` — `fetchPhotos` (~line 1233)

```ts
VideoThumbnails.getThumbnailAsync(item.signedUrl, { time: 0 })
```

To draw a video's grid cell, each device fetches the remote video (enough of it to
decode a frame — over HTTP this is often a large range, and the OS may pull the whole
file) and decodes it locally. Every member pays this per video; the 6-hour
`videoThumb:` cache only helps within one app session (the cache is in-memory), so it
recurs on every cold start. On web there's no thumbnail at all (the block is
native-only).

Meanwhile **`media.thumbnail_key` exists in the schema and is never written.**

**Recommendation:** generate the thumbnail **once, at upload time**, from the local file
(`VideoThumbnails.getThumbnailAsync(localUri)` — no network), upload the JPEG alongside
the video, and store it in `thumbnail_key`. `fetchPhotos` already batch-signs main+alt
keys in one `createSignedUrls` call — fold thumbnail keys into the same batch and use
`transformMediaUrl` on them like photo thumbs. Old rows without a `thumbnail_key` keep
the current client-side fallback.

**Measure:** network inspector on a capsule with a few 20–30 MB videos: grid render
today transfers tens of MB; after, ~30 KB per video. Also fixes web (thumbnails appear)
and removes the main-thread video decode jank when a capsule with several videos opens.

## 🔴 3. Multi-capsule upload re-sends the same bytes once per capsule

**File:** `src/screens/app/PreviewScreen.tsx` — `upload()` (~line 192)

```ts
for (const id of ids) {
  for (let idx = 0; idx < items.length; idx++) {
    await uploadToSingle(id, item.uri, ...);   // full re-upload per capsule
  }
}
```

Selecting 3 capsules for 5 photos performs **15 full uploads** of the same 5 files —
device bandwidth and wall time scale as `capsules × media`.

**Recommendation:** upload each file **once**, then fan out with server-side
`supabase.storage.from('capsule-media').copy(srcKey, destKey)` for the other capsules
(bucket-internal copy, no device bytes), inserting a `media` row per capsule as today.
Note the storage INSERT RLS policy validates the *destination* path's capsule
membership, so copies should be permitted for the same capsules the user could upload
to — verify on a staging bucket first; if `copy` is blocked by policy, an edge
function/RPC with the service role is the fallback.

**Measure:** wall-clock for "5 photos → 3 capsules" drops ~3×; device data usage drops
from `N×M` to `M` file transfers.

Two smaller wins in the same file, same flavor:
- `uploadToSingle` uploads the dual-photo `altUri` **after** the main upload completes —
  `Promise.all` the pair (≈2× on swappable dual photos).
- Both this loop and `CapsuleDetailScreen.uploadPhotos` are strictly sequential per
  file. NSURLSession parallelizes well; 2–3 concurrent uploads roughly halves multi-file
  wall time without starving the connection.

## 🔴 4. Three cron jobs invoke edge functions every minute unconditionally (~130K invocations/month at idle)

**Live `cron.job` rows** (verified): `unlock-capsules`, `close-superlative-windows`, and
`create-group-capsules` each fire `net.http_post` to their edge function `* * * * *`,
regardless of whether any work exists. That's ~43K invocations/month **each** — the
dominant edge-function usage for a pre-launch app, and each call is a cold-start +
4 table scans (unlock does 1 unlock-update + 3 reminder-tier updates per tick).

**Recommendation:** gate each `net.http_post` behind a cheap `EXISTS` in the cron
command, e.g. for unlock-capsules:

```sql
select net.http_post(...)
where exists (
  select 1 from public.capsules
  where status = 'active' and unlock_mode = 'time'
    and unlock_at <= now() + interval '24 hours'   -- covers unlocks AND the widest reminder tier
);
```

Same pattern for the superlative push dispatcher (`exists (select 1 from notifications
where pushed_at is null and type in (...))`) and group-capsule creation (`exists (select 1
from groups where next_capsule_at <= now())`). The SQL-function calls
(`dispatch_superlative_closing_soon`, `close_superlative_windows`) can stay
unconditional — they're in-database and cheap.

**Measure:** edge-function invocation counts on the dashboard drop from ~4,300/day to
near zero at idle. Zero behavior change — the same ticks that would have done work still
fire.

## 🟠 5. `fetchPhotos` fires the `capsule_media_count` RPC on every call, even full cache hits

**File:** `src/screens/app/CapsuleDetailScreen.tsx` — `fetchPhotos` (~line 1151)

The RPC exists solely for the case where RLS hides the media rows (locked
surprise-mode). But it runs unconditionally at the top of `fetchPhotos` — including
warm-cache focuses where the media rows themselves are readable and `mediaData.length`
*is* the count. That's one extra network round-trip on every screen focus of every
capsule, for data that's usually redundant.

**Recommendation:** call the RPC only when the row-read can't be trusted (capsule locked
and — for the owner — `owner_preview_locked`); otherwise set `mediaCount` from the
fetched/cached rows.

**Measure:** one fewer round-trip per CapsuleDetail focus (typically 80–200 ms on
cellular); visible in the network log as the `capsule_media_count` call disappearing on
unlocked capsules.

## 🟠 6. `listMyGroups` / `getGroup` make two serial round-trips where one query works

**File:** `src/lib/groups.ts` (~lines 27–70)

`listMyGroups` fetches memberships, **then** fetches all member rows for those groups
just to count them client-side (transferring one row per member per group). `getGroup`
repeats the pattern. This is on the Home screen's critical path (`useCachedFetch('groups')`).

**Recommendation:** PostgREST embedded aggregates do this in one query with no member
payload:

```ts
supabase.from('group_members')
  .select('group_id, groups(id, name, ..., group_members(count))')
  .eq('user_id', me);
```

**Measure:** Home's group fetch goes from 2 serial round-trips to 1, and payload stops
scaling with total member count.

## 🟠 7. Mutual-capsules lookup ships the caller's entire membership list back to the server

**File:** `src/screens/app/PublicProfileScreen.tsx` — `load()` (~lines 207–224)

Fetches **all** of my `capsule_members` rows, then issues a second query with
`.in('capsule_id', myCapsuleIds)` — the ID list is serialized into the request URL, so
it grows linearly with my membership count (100 memberships ≈ 4 KB URL) and the second
query can't start until the first returns.

**Recommendation:** one query using the server-side membership the RLS layer already
computes — e.g. filter the *other* user's memberships and let the `capsules` embed's own
SELECT policy (membership-gated) do the intersection, or a small
`mutual_capsules(p_other uuid)` RPC. Either collapses 2 serial round-trips into 1 with a
constant-size request.

## 🟠 8. Notifications query is unbounded

**File:** `src/screens/app/NotificationsScreen.tsx` (~line 123)

The unread-notifications select has **no `.limit()`**. Unread rows accumulate for
passive users (nothing auto-reads reaction/suggestion notifications), so a dormant
account returning after months can pull hundreds of rows plus their embedded
capsule/actor joins in one request.

**Recommendation:** `.limit(100)` (the UI groups reactions anyway, so the visible list
is far shorter). Pair with the index in #9.

## 🟡 9. Index gaps for the app's actual query shapes (cheap insurance — tables are small today)

Verified against live `pg_indexes` + Supabase performance advisors:

| Query (frequency) | Today | Recommendation |
|---|---|---|
| Alerts list + tab-badge count: `user_id = ? AND read_at IS NULL ORDER BY sent_at DESC` (every tab switch, most-frequent query in the app) | btree `(user_id)` only | partial `(user_id, sent_at DESC) WHERE read_at IS NULL` |
| Unlock cron: 4 updates/minute filtering `status='active' AND unlock_mode='time' AND unlock_at <= …` | no index touches these columns → 4 seq scans/min, 5,760/day | partial `(unlock_at) WHERE status = 'active' AND unlock_mode = 'time'` |
| Group's capsule list + group deletes | `capsules.group_id` FK unindexed (advisor: `unindexed_foreign_keys`) | index `capsules(group_id)` |
| CapsuleDetail media list: `capsule_id = ? ORDER BY uploaded_at DESC` | `(capsule_id)` only — sort per query | replace with `(capsule_id, uploaded_at DESC)` |
| Notifications embed of `actor` + user-delete cascades | `notifications.actor_id` unindexed (advisor) | index if/when account deletion ships |

The advisors also flag ~10 **unused** indexes (`idx_media_uploader_id`,
`idx_super_votes_voter`, etc.). Don't drop them yet — "unused" at near-zero traffic
mostly means the planner picks seq scans on tiny tables; revisit after real load exists.

## 🟡 10. Expo push requests aren't chunked (correctness-at-scale with perf flavor)

**Files:** `supabase/functions/unlock-capsules/index.ts` (~line 155),
`send-superlative-pushes/index.ts` (~line 83)

Both POST the entire `messages` array to `exp.host` in one request. Expo's API caps a
request at **100 messages** — beyond that the whole batch is rejected, i.e. one popular
capsule unlocking could silently drop *all* pushes for that tick. Chunk into ≤100-message
requests (and `unlock-capsules`' per-capsule `pushTokensFor` calls can be a single
`.in('capsule_id', ids)` query instead of one query per unlocked capsule).

## 🟡 11. Smaller client-side items (each a few minutes, none urgent)

- **Tab badge** (`AppNavigator.CustomTabBar`): issues a fresh count query on tab
  switches (60 s throttle). The Alerts screen already fetches the full unread list into
  the `notifications` cache key — deriving the badge from that cache (subscribe +
  compute) removes the query entirely *and* fixes the stale-badge lag already noted in
  BUGS.md #10.
- **`ProfileScreen` stats**: 4 parallel queries per cold load. Fine as-is (they're
  parallel); a single `profile_stats()` RPC is only worth it if the profile becomes a
  more frequent destination.
- **Web upload path** buffers whole files in memory (`fetch → arrayBuffer`) — fine for
  photos, will hurt for large videos on low-memory devices; consider `Blob` streaming
  upload if web video upload becomes a real path.
- **`HomeScreen` `CountdownBadge`**: one 60 s `setInterval` per card. Negligible now;
  if Home ever holds 100+ cards, lift to a single shared ticker.

## Verified OK (looked for problems, found none)

- **RLS helper functions are planner-friendly**: `get_my_capsule_ids`,
  `get_my_group_ids`, `is_group_creator`, `_superlative_target_valid` are all `STABLE`
  (checked `pg_proc.provolatile`), and policies use `(select auth.uid())` — no
  per-row re-evaluation of auth or volatile helpers on hot read paths.
  (`can_insert_capsule_member` is `VOLATILE`, but it's insert-only — irrelevant.)
- **Signed-URL handling is already batched and cached well**: one `createSignedUrls`
  call for main+alt keys, 50-min cache under the 1-h validity, media rows cached
  separately so warm focuses skip the DB read entirely.
- **`useCachedFetch`** dedupes concurrent fetches via the in-flight registry and skips
  the network on fresh cache hits — the cross-screen fetch architecture is sound.
- **Thumbnails** use the storage render API (`transformMediaUrl`/`transformAvatarUrl`)
  rather than full-res downloads for grids/avatars — correct pattern, keep it.
- **Native uploads** already use `FileSystem.uploadAsync` (bytes never cross the JS
  bridge) with `getFreshAccessToken()` — the transport layer is right; the problem is
  purely the *size* of what's sent (#1).
- **`ProfileScreen` stats** already use `count: 'exact', head: true` (no row payloads).
- **Entrance animations** replay only once per screen instance (`hasAnimatedRef`) — no
  perceived-latency regression on re-focus.

## Suggested order of attack

1. **#1 picker/share resize** — one shared helper, three call sites, biggest single UX win.
2. **#2 upload-time video thumbnails** — schema column already exists; kills the worst
   bandwidth pathology.
3. **#4 cron EXISTS gates** — pure SQL, 10 minutes, ~130K invocations/month saved.
4. **#9 the two hot indexes** (notifications partial, capsules cron partial) + **#8 limit**.
5. **#3 storage-copy fan-out**, then **#5–#7** as touch-the-file-anyway cleanups.
